Chapter 03

The Hidden Costs of LLM Calls in Production

TL;DR

  • At 50k agent runs/day, naive LLM calling costs $15k-$60k/month. Three patterns cut it by 10x.
  • Semantic caching: identical or near-identical questions return cached answers without hitting the LLM.
  • Model routing: classify query complexity and route simple ones to cheap models (GPT-3.5 / Haiku).
  • Context budgeting: rolling window summarisation keeps token counts bounded across long agent runs.
  • The compounding effect: all three together reduce cost per query from $0.12 to under $0.01.

The math that should scare you

Let's be explicit. Your Engineering Wiki Agent handles queries for 3,000 engineers. Assume each engineer asks 15 questions per day. That is 45,000 queries/day.

Each query: 1,500 tokens input (system prompt + retrieved context) + 400 tokens output = ~1,900 tokens. At GPT-4o pricing ($0.005/1K input, $0.015/1K output):

Per query:  (1.5 x $0.005) + (0.4 x $0.015) = $0.0075 + $0.006 = $0.0135
Per day:    45,000 x $0.0135 = $607.50
Per month:  $18,225

That is before retries, eval calls, and agent tool-use reasoning steps which add a 2-5x multiplier. Real monthly spend: $40k-$90k. At that scale, optimisation is survival.

Token cost scales linearly with queries but context window size scales with agent complexity. The two compound each other.

Optimisation 1: Semantic caching

60-70% of questions in any engineering org are near-duplicates. "How do I restart the payment service?" and "What is the command to restart payment-svc?" are semantically identical. Semantic caching embeds every incoming query, searches a vector cache for similar past queries, and returns the cached answer if similarity exceeds a threshold - skipping the LLM entirely.

# semantic_cache.py
import redis, json, numpy as np
from openai import OpenAI

client = OpenAI()
r = redis.Redis(host="localhost", port=6379, decode_responses=True)

CACHE_THRESHOLD = 0.92
CACHE_TTL_S     = 86400  # 24h

def embed(text):
    return client.embeddings.create(model="text-embedding-3-small", input=text).data[0].embedding

def cosine(a, b):
    a, b = np.array(a), np.array(b)
    return float(np.dot(a,b) / (np.linalg.norm(a) * np.linalg.norm(b)))

def cache_lookup(query):
    q_emb = embed(query)
    for key in r.scan_iter("cache:emb:*"):
        cached_emb = json.loads(r.get(key))
        if cosine(q_emb, cached_emb) >= CACHE_THRESHOLD:
            return r.get(key.replace("cache:emb:", "cache:ans:"))
    return None

def cache_store(query, answer):
    import hashlib
    h = hashlib.sha256(query.encode()).hexdigest()[:16]
    r.setex(f"cache:emb:{h}", CACHE_TTL_S, json.dumps(embed(query)))
    r.setex(f"cache:ans:{h}", CACHE_TTL_S, answer)

def query_with_cache(agent, question):
    cached = cache_lookup(question)
    if cached:
        return cached           # LLM call skipped
    result = agent.run(question)
    cache_store(question, result.answer)
    return result.answer

A 65% cache hit rate on 45k daily queries saves 29,250 LLM calls/day - roughly $390/day at GPT-4o pricing.

Optimisation 2: Model routing by complexity

Not every query needs GPT-4o. Route simple factual lookups to fast, cheap models; route complex multi-step reasoning to the frontier model.

# model_router.py
from openai import OpenAI
client = OpenAI()

ROUTER_PROMPT = """Classify this query into one complexity tier:
SIMPLE   - factual lookup, single-step
MODERATE - multi-step reasoning, some synthesis
COMPLEX  - multi-document synthesis, code generation, expert analysis
Respond with only the tier name. Query: {query}"""

MODEL_MAP = {
    "SIMPLE":   "gpt-3.5-turbo",
    "MODERATE": "gpt-4o-mini",
    "COMPLEX":  "gpt-4o",
}

def route(query):
    tier = client.chat.completions.create(
        model="gpt-3.5-turbo",
        messages=[{"role":"user","content": ROUTER_PROMPT.format(query=query)}],
        max_tokens=10, temperature=0,
    ).choices[0].message.content.strip().upper()
    return MODEL_MAP.get(tier, "gpt-4o-mini")

Optimisation 3: Context window budgeting

As agents accumulate conversation history, input token count grows with every turn. A 20-step agent with full history can hit 40k+ tokens per LLM call.

# context_budget.py
import tiktoken
enc = tiktoken.encoding_for_model("gpt-4o")
MAX_CONTEXT_TOKENS = 12_000

def count_tokens(messages):
    return sum(len(enc.encode(m["content"])) for m in messages)

def trim_messages(messages, system):
    system_tok = len(enc.encode(system))
    budget = MAX_CONTEXT_TOKENS - system_tok - 500

    recent  = messages[-4:]    # always keep last 4 verbatim
    history = messages[:-4]

    if history and count_tokens(history) > budget * 0.4:
        summary = summarise_history(history)   # one cheap LLM call
        history = [{"role":"assistant","content":f"[Prior context]: {summary}"}]

    trimmed = history + recent
    while count_tokens(trimmed) > budget and len(trimmed) > 4:
        trimmed.pop(0)

    return [{"role":"system","content":system}] + trimmed

⚠ Common Mistakes & Gotchas

  1. Cache threshold too low (e.g. 0.80). You will return wrong answers for similar but factually different questions. Start at 0.92 and measure the hit rate before lowering.
  2. Routing with the same model you are routing to. Using GPT-4o to decide whether to use GPT-4o defeats the purpose. The router should always be your cheapest model.
  3. Not invalidating the cache on doc updates. Your wiki changes daily. Tie cache TTL to your wiki update frequency, or tag cache entries by document version.
  4. Counting only output tokens. Input tokens dominate volume. A 1,500-token system prompt x 45,000 queries = 67.5M input tokens/day.
  5. Forgetting embedding costs. At high volume, use a local embedding model (sentence-transformers) for cache lookups to make them near-free.
