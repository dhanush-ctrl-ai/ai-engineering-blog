Chapter 09

Prompt Engineering at Scale

TL;DR

  • A prompt is production code. It needs versioning, testing, and a deployment pipeline.
  • Day One: prompt strings hardcoded in your Python files. Fast to change, impossible to track.
  • Production: a prompt registry with semantic versioning, A/B testing via feature flags, regression-safe deploys, and automatic rollback on quality degradation.
  • The most dangerous moment: changing a prompt without running your eval harness first. A 3-word change can silently break 40% of your test cases.
  • Prompt templating with strict variable typing catches interpolation errors before they reach the LLM.

Why prompts need the same discipline as code

In traditional software, a config change goes through code review, CI, and canary deployment before reaching 100% of traffic. A prompt change in most AI teams? Someone edits a string in agent.py, pushes to main, and it is live instantly with no visibility into what changed or whether quality degraded.

Prompts are hyperparameters that run in production. They control model behaviour, output format, safety guardrails, and tool-calling patterns. Treating them as comments in your code is the fastest way to have a silent incident.

A model update with no prompt change can cause a regression. A prompt change with no eval run guarantees one eventually. The two variables must be tested together.

Day One: hardcoded prompt strings

# agent.py - prompt buried in code, no versioning
SYSTEM_PROMPT = """You are an AI assistant for an engineering wiki.
Answer questions based on the retrieved context.
If you don't know, say so. Be concise."""

def run(query: str, context: str) -> str:
    messages = [
        {"role":"system","content":SYSTEM_PROMPT},
        {"role":"user","content":f"Context:\n{context}\n\nQuestion: {query}"}
    ]
    return client.chat.completions.create(model="gpt-4o", messages=messages).choices[0].message.content

The problems: the prompt is coupled to the code deployment cycle, there is no history of what changed and when, you cannot A/B test two versions simultaneously, and rolling back requires a code deploy.

Production: a prompt registry

A prompt registry treats prompts as versioned artefacts stored separately from code. Think of it as a database table for prompts: each row is a version with a semantic version number, the template text, metadata, and eval results. Deploying a prompt is a database operation, not a git push.

# prompts/registry.py
from dataclasses import dataclass
from datetime import datetime
import postgres  # or any DB client

@dataclass
class PromptVersion:
    name: str             # "wiki_agent_system"
    version: str          # "2.3.1" - semver
    template: str         # the actual prompt with {variable} slots
    variables: list[str]  # required variables: ["context", "query"]
    model: str            # intended model: "gpt-4o"
    eval_score: float     # faithfulness score at deploy time: 0.91
    deployed: bool        # is this the active version?
    created_at: datetime

class PromptRegistry:
    def get_active(self, name: str) -> PromptVersion:
        return db.query_one(
            "SELECT * FROM prompts WHERE name=%s AND deployed=true", name
        )

    def render(self, name: str, **kwargs) -> str:
        prompt = self.get_active(name)
        # Validate all required variables are provided
        missing = [v for v in prompt.variables if v not in kwargs]
        if missing:
            raise ValueError(f"Missing prompt variables: {missing}")
        # Validate no extra variables (prevent injection via template)
        extra = [k for k in kwargs if k not in prompt.variables]
        if extra:
            raise ValueError(f"Unknown prompt variables: {extra}")
        return prompt.template.format(**kwargs)

    def deploy(self, name: str, version: str, min_eval_score: float = 0.85) -> bool:
        candidate = db.query_one(
            "SELECT * FROM prompts WHERE name=%s AND version=%s", name, version
        )
        if candidate.eval_score < min_eval_score:
            raise ValueError(
                f"Cannot deploy: eval score {candidate.eval_score:.2f} < threshold {min_eval_score}"
            )
        # Atomic swap: deactivate current, activate new
        with db.transaction():
            db.execute("UPDATE prompts SET deployed=false WHERE name=%s AND deployed=true", name)
            db.execute("UPDATE prompts SET deployed=true WHERE name=%s AND version=%s", name, version)
        return True

registry = PromptRegistry()

A/B testing prompt versions

# prompts/ab_test.py
import hashlib
from feature_flags import get_flag  # LaunchDarkly, Statsig, etc.

def get_prompt_for_user(prompt_name: str, user_id: str) -> PromptVersion:
    """Route users to prompt variants based on feature flag bucketing."""
    # Deterministic bucket: same user always gets same variant
    bucket = int(hashlib.md5(user_id.encode()).hexdigest(), 16) % 100

    # Feature flag defines: {"control": "2.3.1", "treatment": "2.4.0-beta", "split": 20}
    flag = get_flag(f"prompt_ab_{prompt_name}")

    if bucket < flag["split"]:
        version = flag["treatment"]
    else:
        version = flag["control"]

    return registry.get_version(prompt_name, version)

# Track which version generated each response for analysis
def log_response(user_id: str, prompt_version: str, quality_score: float):
    analytics.track("prompt_response", {
        "user_id": user_id,
        "prompt_version": prompt_version,
        "quality_score": quality_score,

Regression-safe deployment pipeline

# ci/deploy_prompt.py - runs in GitHub Actions on prompt file changes
import sys
from prompts.registry import PromptRegistry
from tests.eval.runner import run_all_evals

async def deploy_prompt_safely(prompt_name: str, new_version: str):
    registry = PromptRegistry()
    current = registry.get_active(prompt_name)
    candidate = registry.get_version(prompt_name, new_version)

    print(f"Evaluating {prompt_name} v{new_version} vs current v{current.version}...")

    # Run full eval suite against candidate version
    results = await run_all_evals(prompt_override=candidate.template)
    new_score = results.avg_faithfulness

    print(f"Current:   faithfulness={current.eval_score:.3f}")
    print(f"Candidate: faithfulness={new_score:.3f}")

    if new_score < current.eval_score - 0.02:   # 2% regression threshold
        print(f"BLOCKED: score regressed by {current.eval_score - new_score:.3f}")
        sys.exit(1)

    if new_score < 0.85:
        print(f"BLOCKED: score {new_score:.3f} below minimum threshold 0.85")
        sys.exit(1)

    # Update eval score in registry, then deploy
    registry.update_eval_score(prompt_name, new_version, new_score)
    registry.deploy(prompt_name, new_version)
    print(f"Deployed {prompt_name} v{new_version} (score: {new_score:.3f})")

Day One vs Production

DimensionDay OneProduction
StorageHardcoded in source codeVersioned DB table / YAML registry
DeploymentCode push requiredDB operation, instant, no deploy
HistoryGit blame (if you're lucky)Full version history with eval scores
A/B testingNot possibleFeature flag bucketing per user
RollbackRevert commit + redeploySet deployed=false, instant
Safety gateNoneEval score must pass before deploy allowed

⚠ Common Mistakes & Gotchas

  1. Changing model and prompt at the same time. If quality changes, you do not know whether it was the model or the prompt. Change one variable at a time and run evals between changes.
  2. No regression threshold on deploy. A prompt that scores 0.91 on a previous eval run can drop to 0.83 with a new version. Gate deploys on absolute score AND delta from current production version.
  3. f-strings instead of explicit variable lists. Python f-strings allow any variable in scope, making injection trivial. Use a template system that enforces an allowlist of variable names.
  4. Prompt A/B tests without enough traffic. You need at least 200-300 responses per variant to detect a meaningful difference in quality scores. Do not conclude a test early based on 20 samples.
  5. Storing prompts in environment variables. Env vars have no history, no atomic swap, and no eval gating. Use a real registry or at minimum a versioned YAML file in a prompts/ directory.
