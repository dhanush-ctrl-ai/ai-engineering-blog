Chapter 08

Memory Systems for Agents

TL;DR

  • Without memory, every agent conversation starts from zero. Engineers repeat context. Quality degrades.
  • Four memory types: in-context (conversation buffer), episodic (past interactions), semantic (facts about users and entities), procedural (learned preferences and workflows).
  • Day One: a rolling message buffer that keeps the last N turns in the context window.
  • Production: mem0-style write-time fact extraction, deduplication, and hybrid retrieval (vector + BM25 + entity graph) at query time.
  • Memory is a database. Design it with the same rigour you would apply to any other production datastore.

The stateless agent problem

Every LLM call is stateless by default. There is no memory between conversations. When an engineer comes back the next day and asks "did that P1 we discussed get resolved?", the agent has no idea what P1 they mean, who they are, or that any conversation happened at all.

The naive fix is to put the entire conversation history in the context window. This works until the history grows beyond the context limit, which at GPT-4o's 128k tokens happens faster than you expect once you include system prompt, retrieved chunks, and tool call history.

A context window is working memory - fast but finite. True agent memory requires a persistent external store that survives across sessions.

The four types of agent memory

Day One: rolling context buffer

# memory/simple.py
from collections import deque

class RollingBuffer:
    def __init__(self, max_messages: int = 20):
        self.buffer = deque(maxlen=max_messages)

    def add(self, role: str, content: str):
        self.buffer.append({"role": role, "content": content})

    def get_messages(self) -> list[dict]:
        return list(self.buffer)

# Usage
memory = RollingBuffer(max_messages=20)
memory.add("user", "How do I restart the payment service?")
memory.add("assistant", "Run: kubectl rollout restart deployment/payment-svc")

# On next turn, history injected into context automatically
messages = [{"role":"system","content":SYSTEM_PROMPT}] + memory.get_messages()
response = client.chat.completions.create(model="gpt-4o", messages=messages)

This works for a single session. It has three hard limits: it forgets everything when the session ends, it blows up the context window with long conversations, and it stores no facts about the user across sessions.

Production: mem0-style persistent memory

The production approach has two phases. At write time, when a conversation ends, an LLM extracts structured facts from the session and stores them deduplicated into a memory store. At read time, before each agent response, relevant memories are retrieved and injected into the context window as a compact memory block.

# memory/production.py
import json
from openai import OpenAI
from qdrant_client import QdrantClient

client = OpenAI()
qdrant = QdrantClient(url="http://localhost:6333")

EXTRACT_PROMPT = """Extract structured facts from this conversation.
Return a JSON array of fact objects. Each fact must have:
- "fact": the factual statement (concise, third-person)
- "category": one of [user_preference, entity_relationship, action_taken, system_state]
- "entities": list of entity names mentioned
- "confidence": 0.0-1.0

Only extract facts that would be useful in future conversations.
Conversation:
{conversation}"""

def extract_facts(conversation: list[dict]) -> list[dict]:
    """Write path: extract structured facts from a finished session."""
    text = "\n".join(f"{m['role'].upper()}: {m['content']}" for m in conversation)
    response = client.chat.completions.create(
        model="gpt-4o-mini",   # cheap model for extraction
        messages=[{"role":"user","content": EXTRACT_PROMPT.format(conversation=text)}],
        response_format={"type":"json_object"},
    )
    data = json.loads(response.choices[0].message.content)
    return data.get("facts", [])

def store_facts(facts: list[dict], user_id: str):
    """Deduplicate and store facts in the memory store."""
    for fact in facts:
        if fact.get("confidence", 0) < 0.7:
            continue  # skip low-confidence extractions
        embedding = client.embeddings.create(
            model="text-embedding-3-small", input=fact["fact"]
        ).data[0].embedding
        # Upsert: if very similar fact exists (sim > 0.95), update instead of duplicate
        qdrant.upsert(collection_name=f"memory_{user_id}", points=[{
            "id": generate_fact_id(fact["fact"]),
            "vector": embedding,
            "payload": {**fact, "user_id": user_id, "created_at": now_iso()}
        }])

def retrieve_memories(query: str, user_id: str, top_k: int = 5) -> str:
    """Read path: retrieve relevant memories as a formatted context block."""
    q_emb = client.embeddings.create(model="text-embedding-3-small", input=query).data[0].embedding
    results = qdrant.search(
        collection_name=f"memory_{user_id}",
        query_vector=q_emb,
        limit=top_k,
        score_threshold=0.72,
    )
    if not results:
        return ""
    facts = [r.payload["fact"] for r in results]
    return "## What I remember about you:\n" + "\n".join(f"- {f}" for f in facts)

def build_context(query: str, user_id: str, system_prompt: str) -> list[dict]:
    memories = retrieve_memories(query, user_id)
    system = system_prompt + ("\n\n" + memories if memories else "")
    return [{"role": "system", "content": system}]

Day One vs Production

DimensionDay OneProduction
PersistenceLost at session endSurvives across sessions and deployments
StorageContext window (max ~80k tokens)Unlimited external store
Memory typeRaw messagesStructured extracted facts
RetrievalAll-or-nothing (full buffer)Relevant-only (vector similarity)
DeduplicationNoneWrite-time merge on high-similarity facts
User isolationNone - one buffer per instancePer-user namespaced collection

⚠ Common Mistakes & Gotchas

  1. Storing raw messages instead of extracted facts. A 50-turn conversation compresses to 8-12 facts. Storing raw messages bloats your vector DB and retrieves irrelevant context. Always extract at write time.
  2. No deduplication. Without deduplication, every session adds new facts even if they contradict old ones. "User prefers Python" stored 12 times is useless and confusing. Check similarity before inserting.
  3. Retrieving too many memories. Injecting 20 memory facts into every prompt is expensive and noisy. Use a similarity threshold (0.72+) and cap at 5-7 facts. Irrelevant memories hurt more than no memories.
  4. Treating memory as infallible. Extracted facts can be wrong. The LLM misinterprets sarcasm, hypotheticals, and quoted speech as facts. Always store confidence and filter below 0.70.
  5. Not expiring stale memories. "Riya owns the payments team" might have been true 6 months ago. Memories need a TTL or a staleness score that degrades over time.
