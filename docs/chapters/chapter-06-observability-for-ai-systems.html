Chapter 06

Observability for AI Systems: Tracing What You Can't See

TL;DR

  • You cannot debug an AI system you cannot see. "It returned a bad answer" is not a diagnosis.
  • Day One: structured JSON logging with query, retrieved chunks, answer, and latency on every request.
  • Production: OpenTelemetry spans for every agent step, Langfuse traces for LLM calls, per-query cost tracking, retrieval quality metrics, and PagerDuty alerts on degradation.
  • The four signals: retrieval relevance score, faithfulness score, p95 latency, and cost per query.
  • Golden debugging path: trace_id -> which chunks were retrieved -> what prompt was sent -> what the LLM returned -> why the answer was wrong.

Why AI systems are hard to observe

Traditional software observability is about did it run. AI system observability is about did it think correctly. Those are fundamentally different problems.

An agent that returns HTTP 200 with a coherent-sounding answer can still be catastrophically wrong. It might have retrieved the wrong runbook. It might have hallucinated a Jira ticket number. It might have answered a question about the 2022 on-call rotation using 2024 data. None of this shows up in your existing APM dashboards.

Your LLM is a black box between your retriever and your user. Observability is the glass you install in the box.

The four signals that matter

Day One: structured logging

The minimum viable observability is structured JSON logs that capture everything you need to reconstruct what happened for any query. No tracing infrastructure needed - just a disciplined log schema.

# observability/logger.py
import json, time, logging
from dataclasses import dataclass, asdict
import tiktoken

enc = tiktoken.encoding_for_model("gpt-4o")
logger = logging.getLogger("agent.query")

@dataclass
class QueryTrace:
    trace_id: str
    query: str
    retrieved_chunks: list     # [{text, score, source_file}]
    prompt_tokens: int
    completion_tokens: int
    answer: str
    latency_ms: float
    cost_usd: float
    retrieval_avg_score: float

def log_query(trace):
    logger.info(json.dumps({"event": "agent.query", **asdict(trace)}))

def traced_query(agent, query, trace_id):
    start = time.perf_counter()
    result = agent.run(query)
    latency_ms = (time.perf_counter() - start) * 1000

    chunks = [
        {"text": n.text[:200], "score": n.score, "source": n.metadata.get("file_name","")}
        for n in result.source_nodes
    ]
    prompt_tok  = len(enc.encode(result.metadata.get("prompt", "")))
    answer_tok  = len(enc.encode(result.answer))
    cost        = (prompt_tok * 0.000005) + (answer_tok * 0.000015)
    avg_score   = sum(n.score for n in result.source_nodes) / max(len(result.source_nodes), 1)

    log_query(QueryTrace(
        trace_id=trace_id, query=query, retrieved_chunks=chunks,
        prompt_tokens=prompt_tok, completion_tokens=answer_tok,
        answer=result.answer, latency_ms=round(latency_ms, 1),
        cost_usd=round(cost, 5), retrieval_avg_score=round(avg_score, 3),
    ))
    return result.answer

Production: OpenTelemetry + Langfuse

Structured logs answer "what happened." Distributed traces answer "why it happened and where time was spent." OpenTelemetry gives you spans for every agent step; Langfuse gives you LLM-specific traces with token counts, costs, and prompt versions.

# observability/tracing.py
from opentelemetry import trace
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.trace.export import BatchSpanProcessor
from opentelemetry.exporter.otlp.proto.grpc.trace_exporter import OTLPSpanExporter
from langfuse import Langfuse
import functools, time

provider = TracerProvider()
provider.add_span_processor(BatchSpanProcessor(OTLPSpanExporter(endpoint="http://otel-collector:4317")))
trace.set_tracer_provider(provider)
tracer = trace.get_tracer("agent.tracer")

langfuse = Langfuse(public_key=LANGFUSE_PUBLIC_KEY, secret_key=LANGFUSE_SECRET_KEY)

def trace_agent_step(step_name):
    def decorator(fn):
        @functools.wraps(fn)
        async def wrapper(*args, **kwargs):
            with tracer.start_as_current_span(step_name) as span:
                start = time.perf_counter()
                try:
                    result = await fn(*args, **kwargs)
                    span.set_attribute("step.success", True)
                    return result
                except Exception as e:
                    span.set_attribute("step.success", False)
                    span.set_attribute("error.message", str(e))
                    raise
                finally:
                    span.set_attribute("step.latency_ms", round((time.perf_counter()-start)*1000, 1))
        return wrapper
    return decorator

@trace_agent_step("retrieve")
async def retrieve_context(query, index):
    return await index.aretrieve(query)

@trace_agent_step("llm_generate")
async def generate_answer(prompt, model):
    generation = langfuse.generation(
        name="answer_generation", model=model,
        input=[{"role":"user","content":prompt}],
    )
    response = await client.chat.completions.acreate(
        model=model, messages=[{"role":"user","content":prompt}]
    )
    answer = response.choices[0].message.content
    generation.end(output=answer, usage={
        "input": response.usage.prompt_tokens,
        "output": response.usage.completion_tokens,
    })
    return answer

Alerting on retrieval degradation

# alerts/retrieval_monitor.py
import statistics
from collections import deque

score_window = deque(maxlen=300)  # rolling 5-min window at 1 query/sec

def record_retrieval_score(avg_score):
    score_window.append(avg_score)

def check_retrieval_health():
    if len(score_window) < 50:
        return True  # not enough data
    window_avg = statistics.mean(score_window)
    if window_avg < 0.72:
        pagerduty.trigger(
            summary=f"RAG retrieval degraded: avg score {window_avg:.2f} (threshold 0.72)",
            severity="warning",
            details={"window_size": len(score_window), "avg_score": window_avg},
        )
        return False
    return True

The golden debugging path

When a user reports a wrong answer, here is the exact sequence to diagnose it in under 5 minutes:

  1. Get the trace_id from the user's session or from your query log.
  2. Pull the Langfuse trace - see exactly what prompt was sent to the LLM, including all retrieved chunks.
  3. Check retrieval scores - if avg_score is below 0.70, the retriever fetched irrelevant chunks. Root cause: embedding model drift or missing documents.
  4. Check faithfulness - if chunks are relevant but the answer is wrong, the LLM is hallucinating. Root cause: prompt template issue or model regression.
  5. Check latency span - if retrieval takes 8s out of 10s total, the bottleneck is the vector DB. Root cause: index growth without HNSW migration.

⚠ Common Mistakes & Gotchas

  1. Logging answers but not retrieved chunks. The answer alone tells you nothing. You need the chunks to know whether the retriever or the LLM is the failure point.
  2. Sampling faithfulness at 1%. At 45k queries/day that is 450 evals - barely enough to detect gradual degradation. Sample at 5% minimum.
  3. No prompt versioning in traces. When you update a prompt template, old traces have the old prompt. Without versioning you cannot tell whether a regression happened before or after the prompt change.
  4. Alerting on individual query latency. Single-query spikes are normal. Alert on p95 over a 5-minute window. A single 30-second query does not mean the system is degraded.
  5. Not correlating retrieval score with answer quality. A low retrieval score is a leading indicator that bad answers are coming. Build a dashboard that plots both together over time.
Observability is not a feature you add later. Every agent step that runs without a trace is a future debugging session you are billing your future self for.
