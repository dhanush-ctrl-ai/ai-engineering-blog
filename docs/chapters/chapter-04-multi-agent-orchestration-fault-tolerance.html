Chapter 04

Multi-Agent Orchestration: Designing for Fault Tolerance

TL;DR

  • Multi-agent pipelines multiply your failure surface: one stuck agent blocks everything downstream.
  • Day One: a sequential pipeline where agent B calls agent A's output directly - simple, brittle.
  • Production: async task queues between agents, per-agent circuit breakers, partial result caching, and compensation logic for side effects.
  • The orchestrator pattern: a supervisor agent that assigns tasks, monitors health, and re-routes on failure.
  • Key insight: agents are microservices. Apply the same fault-tolerance patterns you already know.

Why multi-agent systems fail differently

A single agent has one failure surface: the LLM call and its tools. A five-agent pipeline has five failure surfaces plus the communication layer between them. Failures compound: if Agent 3 fails after Agents 1 and 2 have already written to external systems, you have partial state in Jira, Confluence, and Slack simultaneously.

A chain is only as reliable as its weakest link. In a 5-agent pipeline with each agent at 99% reliability, the pipeline succeeds only 95% of the time. At 95% per agent: 77% success rate.

The running example: Incident Response Pipeline

When an alert fires, a five-agent pipeline kicks off: triage the severity, find the runbook, create a Jira ticket, notify Slack, and update the Confluence status page.

Day One: sequential pipeline

# pipeline_v1.py - simple, breaks on any agent failure
def run_incident_pipeline(alert):
    severity = triage_agent.run(alert)
    runbook  = search_agent.run(alert, severity)
    ticket   = jira_agent.run(alert, runbook)     # if this fails, steps 4+5 never run
    slack_agent.run(alert, ticket)
    confluence_agent.run(alert, ticket)
    return {"ticket_id": ticket.id}

Production: the orchestrator pattern

The orchestrator maintains a task graph, dispatches work to stateless sub-agent workers via a queue, and propagates failures gracefully by marking dependents as SKIPPED rather than crashing.

# orchestrator.py
import asyncio
from dataclasses import dataclass, field
from enum import Enum

class TaskStatus(Enum):
    PENDING = "pending"
    RUNNING = "running"
    DONE    = "done"
    FAILED  = "failed"
    SKIPPED = "skipped"

@dataclass
class AgentTask:
    id: str
    agent_type: str
    input: dict
    depends_on: list = field(default_factory=list)
    status: TaskStatus = TaskStatus.PENDING
    result: dict = None
    retries: int = 0
    max_retries: int = 3

class Orchestrator:
    def __init__(self, workers):
        self.workers = workers
        self.tasks = {}
        self.results = {}

    async def run(self):
        while not self._all_terminal():
            ready = self._get_ready_tasks()
            if ready:
                await asyncio.gather(*[self._execute(t) for t in ready])
            else:
                await asyncio.sleep(0.1)
        return self.results

    def _all_terminal(self):
        terminal = {TaskStatus.DONE, TaskStatus.FAILED, TaskStatus.SKIPPED}
        return all(t.status in terminal for t in self.tasks.values())

    def _get_ready_tasks(self):
        ready = []
        for task in self.tasks.values():
            if task.status != TaskStatus.PENDING:
                continue
            deps_done = all(self.tasks[d].status == TaskStatus.DONE for d in task.depends_on)
            if deps_done:
                ready.append(task)
        return ready

    async def _execute(self, task):
        task.status = TaskStatus.RUNNING
        worker = self.workers[task.agent_type]
        try:
            result = await worker.run(task.input, self.results)
            task.result = result
            task.status = TaskStatus.DONE
            self.results[task.id] = result
        except Exception as e:
            task.retries += 1
            if task.retries <= task.max_retries:
                task.status = TaskStatus.PENDING   # re-queue
            else:
                task.status = TaskStatus.FAILED
                self._skip_dependents(task.id)

    def _skip_dependents(self, failed_id):
        for task in self.tasks.values():
            if failed_id in task.depends_on and task.status == TaskStatus.PENDING:
                task.status = TaskStatus.SKIPPED
                self._skip_dependents(task.id)   # cascade

Compensation: undoing partial work

# If Jira agent succeeds but Slack agent fails, delete the orphaned ticket.
COMPENSATIONS = {
    "jira_task":       ("delete_jira_ticket", lambda r: {"ticket_id": r["id"]}),
    "confluence_task": ("revert_confluence",  lambda r: {"page_id":   r["page_id"]}),
}

async def compensate(orchestrator):
    done_tasks = [t for t in orchestrator.tasks.values()
                  if t.status == TaskStatus.DONE and t.id in COMPENSATIONS]
    for task in reversed(done_tasks):
        comp_fn, args_fn = COMPENSATIONS[task.id]
        await orchestrator.workers["compensator"].run(
            {"action": comp_fn, "args": args_fn(task.result)}
        )

⚠ Common Mistakes & Gotchas

  1. Cascading failures without SKIPPED state. If a failed agent's dependents are left as PENDING forever, your orchestrator loops. Always propagate failure to dependents explicitly.
  2. Infinite retry loops. An agent retrying indefinitely starves the queue. Always enforce max_retries and move to FAILED after exhaustion.
  3. No partial result caching. If a 5-agent pipeline fails at step 4, rerunning from step 1 wastes money. Cache each agent's output by run_id + task_id and resume from the checkpoint.
  4. Compensation without idempotency. If compensation itself fails halfway through, you need idempotent compensation actions. Use the idempotency-key pattern from Chapter 01.
  5. Treating all failures as transient. A Jira 401 auth error should not be retried 3 times. Classify errors before retrying.
