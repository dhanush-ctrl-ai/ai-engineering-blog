Chapter 01

Building a Production AI Agent with Failure Recovery

TL;DR

  • The naive agent loop crashes at 2 AM. Every production incident traces back to the same four root causes: no retries, no timeouts, no dead-letter queue, no compensation logic.
  • Day One: a synchronous while-loop with direct LLM calls. Works for demos, breaks under load.
  • Production: Celery + Redis async task queue, exponential backoff, circuit breakers (pybreaker), idempotency keys, failure classification, and saga-pattern compensation.
  • The ReAct loop is not a reliability mechanism. It is a reasoning pattern. You must add the reliability yourself.
  • Every agent action that touches an external system must be idempotent. No exceptions.

The two-tier mental model

Before writing any code, you need two mental models running in parallel. The first is the reasoning model: how the agent thinks, which tools it selects, how it processes observations. The second is the reliability model: what happens when a tool call times out at step 4 of 7, when the LLM returns a malformed tool call JSON, when Jira is down during an incident response.

Most tutorials cover only the first model. Production systems require both.

A ReAct loop is a pattern for multi-step reasoning. It has no opinion on retries, circuit breakers, or dead-letter queues. Those are your job.

The ReAct loop internals

ReAct (Reasoning + Acting) alternates between two operations: Thought (the LLM reasons about what to do next) and Action (your code executes a tool and returns an Observation). The loop continues until the LLM emits a final answer or a configured step limit is hit.

Day One: the naive loop

# agent_v1.py - works for demos, breaks in production
from openai import OpenAI
import json

client = OpenAI()

def run_agent(query: str, tools: list, tool_map: dict) -> str:
    messages = [{"role": "user", "content": query}]
    
    for step in range(10):  # max 10 steps
        response = client.chat.completions.create(
            model="gpt-4o", messages=messages,
            tools=tools, tool_choice="auto"
        )
        msg = response.choices[0].message
        
        if msg.tool_calls:
            messages.append(msg)
            for call in msg.tool_calls:
                args = json.loads(call.function.arguments)
                result = tool_map[call.function.name](**args)  # no timeout, no retry
                messages.append({
                    "role": "tool", "tool_call_id": call.id,
                    "content": str(result)
                })
        else:
            return msg.content  # final answer
    
    return "Max steps reached"

This breaks in four ways: a hung Jira API call blocks the thread forever, a transient 429 rate limit causes a hard failure, a Jira ticket created before a later step fails leaves orphaned state, and two concurrent requests with the same query create duplicate tickets.

Production: the hardened runtime

1. Async task queue with Celery + Redis

# tasks/agent_task.py
from celery import Celery
from kombu import Queue
import uuid

app = Celery('agent', broker='redis://localhost:6379/0', backend='redis://localhost:6379/0')

app.conf.task_queues = [
    Queue('agent_runs',   routing_key='agent.run'),
    Queue('dead_letter',  routing_key='agent.dead'),
]
app.conf.task_acks_late = True           # ack only after success - safe re-queue on crash
app.conf.task_reject_on_worker_lost = True

@app.task(
    bind=True,
    max_retries=3,
    default_retry_delay=5,
    queue='agent_runs',
    time_limit=120,          # hard kill after 2 min
    soft_time_limit=90,      # SoftTimeLimitExceeded raised at 90s
)
def run_agent_task(self, query: str, run_id: str = None):
    run_id = run_id or str(uuid.uuid4())
    try:
        return _execute_agent(query, run_id)
    except SoftTimeLimitExceeded:
        raise self.retry(exc=SoftTimeLimitExceeded("Agent timed out"), countdown=10)
    except Exception as exc:
        raise self.retry(exc=exc, countdown=2 ** self.request.retries)  # exponential backoff

2. Circuit breaker per tool

# reliability/circuit_breaker.py
import pybreaker

# Each external service gets its own breaker
jira_breaker  = pybreaker.CircuitBreaker(fail_max=3, reset_timeout=60)
slack_breaker = pybreaker.CircuitBreaker(fail_max=5, reset_timeout=30)

def call_jira(fn, *args, **kwargs):
    try:
        return jira_breaker.call(fn, *args, **kwargs)
    except pybreaker.CircuitBreakerError:
        # Return structured error instead of crashing the agent
        return {"error": "Jira unavailable", "retry_after": 60}

# State transitions:
# CLOSED (normal) -> OPEN (fail_max exceeded) -> HALF_OPEN (reset_timeout elapsed)
# HALF_OPEN: one trial request. Success -> CLOSED. Fail -> OPEN again.

3. Idempotency keys - the most important pattern

# reliability/idempotency.py
import hashlib, redis

r = redis.Redis(host='localhost', port=6379)
TTL = 86400  # 24 hours

def make_key(run_id: str, step: int, tool_name: str) -> str:
    raw = f"{run_id}:{step}:{tool_name}"
    return f"idempotency:{hashlib.sha256(raw.encode()).hexdigest()[:16]}"

def idempotent_call(run_id: str, step: int, tool_name: str, fn, *args, **kwargs):
    key = make_key(run_id, step, tool_name)
    
    # Check if we already ran this exact step
    cached = r.get(key)
    if cached:
        return json.loads(cached)   # return prior result, skip the call
    
    result = fn(*args, **kwargs)    # execute the tool
    r.setex(key, TTL, json.dumps(result))  # cache the result
    return result

# Now if the agent retries after a timeout, step 3 (Jira ticket creation)
# returns the existing ticket instead of creating a duplicate.

4. Failure classification - not all errors are the same

# reliability/failure_classifier.py
from enum import Enum

class FailureType(Enum):
    TRANSIENT  = "transient"   # retry immediately: 429, 503, network blip
    RETRYABLE  = "retryable"   # retry with backoff: 500, timeout
    PERMANENT  = "permanent"   # do not retry: 401, 400, 404
    UNKNOWN    = "unknown"

def classify(exc: Exception) -> FailureType:
    if hasattr(exc, 'status_code'):
        code = exc.status_code
        if code in (429, 503, 502): return FailureType.TRANSIENT
        if code in (500, 504):      return FailureType.RETRYABLE
        if code in (400, 401, 403, 404): return FailureType.PERMANENT
    if "timeout" in str(exc).lower(): return FailureType.RETRYABLE
    return FailureType.UNKNOWN

def should_retry(exc: Exception, attempt: int) -> bool:
    ft = classify(exc)
    if ft == FailureType.PERMANENT: return False    # never
    if ft == FailureType.TRANSIENT: return attempt < 5
    if ft == FailureType.RETRYABLE: return attempt < 3
    return attempt < 2

⚠ Common Mistakes & Gotchas

  1. Retrying permanent errors. A 401 Unauthorized from Jira will not fix itself with retries. Classify errors before retrying or you will spam the external service and burn your retry budget on futile attempts.
  2. Idempotency keys that are too coarse. Using just the run_id as the key means retrying step 1 skips steps 2-7 as well. Key must be (run_id, step_index, tool_name) to allow partial retry from the failed step.
  3. No dead-letter queue. Tasks that exhaust max_retries must go somewhere. Without a DLQ, you lose visibility into chronic failures entirely. Always configure a DLQ and alert on it.
  4. Circuit breaker shared across all callers. If the Jira circuit breaker is a global singleton, a single degraded endpoint opens the breaker for all users simultaneously. Scope breakers per-service, not per-call-site.
  5. Compensation logic that is not idempotent. If your rollback itself fails halfway through and is retried, non-idempotent compensation creates new corrupted state. Apply the same idempotency key pattern to compensation actions.
