Building a Production AI Agent
with Failure Recovery

From a naive while-loop that crashes at 2 AM, to a hardened agent runtime with circuit breakers, dead-letter queues, and compensation logic — the backend engineer's guide to agents that actually stay up.

Dhanush · June 2026 · ~14 min read · Advanced
TL;DR

  • Day One: a dead-simple agent loop with try/except. Good for demos, dead on arrival in prod.
  • Hyper-Growth: async task queues (Celery/SQS), circuit breakers, dead-letter queues, idempotent tool calls, and structured retry budgets.

01

Why This Actually Matters

Let's be honest about something. The AI agent tutorial ecosystem has a dirty secret: every demo runs perfectly because every demo is written to run perfectly. The LLM call always succeeds. The tool always returns a result. The agent always terminates cleanly.

Real production looks like this: your agent is halfway through a 10-step workflow — it has fetched data, written a draft, called two external APIs — and then the LLM times out. What happens? If you followed a tutorial, the answer is: you crash, you lose all intermediate state, and you silently drop the task. The user gets nothing. Your on-call engineer gets paged at 2 AM.

"An agent is not a script. It's a distributed system with LLMs as one of the compute nodes. And distributed systems fail."

This is where backend engineers have a massive advantage over pure ML practitioners. You already know about circuit breakers, exponential backoff, idempotency keys, dead-letter queues, and compensation transactions. You've felt the pain of a Celery task that retried itself into an infinite loop. You've debugged a race condition at 3 AM.

This chapter translates all of that hard-won production intuition into the agentic AI world. We'll go from a naive agent loop to a fault-tolerant agent runtime — and we'll be explicit about every trade-off.

Running use case

Throughout this chapter, we're building an Engineering Wiki Agent — the same system from our RAG series, but now it can autonomously take actions: search the wiki, create Jira tickets, send Slack alerts, and update Confluence pages. It processes ~50,000 agent runs per day across a 3,000-person engineering org. That's the scale we're designing for.

02

Internals: How Agent Loops Work

Before we harden anything, we need to understand what we're hardening. Strip every AI agent framework down to its core and you get the same basic structure — a ReAct loop: Reason, Act, Observe, repeat.

Fig 1 — The ReAct agent loop. Failures can occur at every arrow.

The critical insight is that every single arrow in this diagram is a network call. The "Think" step calls the LLM API — that can fail or time out. The "Act" step calls an external tool — Jira might be down, your RAG service might be rate-limited, Slack might return a 503. The "Observe" step might get back a malformed response the agent can't parse.

In a naive implementation, any of these failures crashes the entire loop and loses all context. In a production system, each failure has a defined recovery path.

03

Day One: The MVP

The metaphor here is a solo contractor. One person, one notebook, one phone. If they get sick, the job stops. That's fine when there's one job. This is your MVP agent — dead simple, good enough to ship, and honest about its limitations.

The naive agent loop

agent_v1.py Python 
# Day One: works fine, will haunt you at scale
import openai
import json

def run_agent(task: str, tools: list, max_steps: int = 10) -> str:
    client = openai.OpenAI()
    messages = [{"role": "user", "content": task}]

    for step in range(max_steps):
        # ⚠ No timeout, no retry, no circuit breaker
        response = client.chat.completions.create(
            model="gpt-4o",
            messages=messages,
            tools=tools,
        )
        msg = response.choices[0].message

        # done?
        if not msg.tool_calls:
            return msg.content

        # execute each tool call (also no error handling)
        for call in msg.tool_calls:
            result = execute_tool(call.function.name, call.function.arguments)
            messages.append({
                "role": "tool",
                "tool_call_id": call.id,
                "content": json.dumps(result),
            })

    return "Max steps reached."  # silent failure 😬


def execute_tool(name: str, args_json: str) -> dict:
    args = json.loads(args_json)
    # ⚠ No timeout, no retry, no idempotency
    if name == "search_wiki":
        return wiki_client.search(args["query"])
    elif name == "create_jira_ticket":
        return jira_client.create(**args)
    raise ValueError(f"Unknown tool: {name}")
What this breaks in production

OpenAI times out after 600s by default. Jira's API rate limits at 300 req/min. If the agent retries create_jira_ticket after a transient failure, it creates a duplicate ticket. If the process restarts mid-run, all context is lost. These are not edge cases — at 50,000 runs/day, they are daily events.

When to ship the MVP anyway

Ship it when you're validating the workflow logic, not the runtime reliability. Internal demos, single-user prototypes, batch jobs that run once a day with a human in the loop — all fine. The moment you have concurrent users or automated triggers, you need the next level.

⚡ Day One
  • Synchronous, blocking loop
  • No retry logic anywhere
  • State lives in memory only
  • Silent failures ("max steps reached")
  • Tool calls are not idempotent
  • No observability or tracing
🚀 Production
  • Async task queue (Celery + Redis)
  • Retry with exponential backoff
  • State persisted in Postgres/Redis
  • Explicit failure states + alerts
  • Idempotency keys on all side effects
  • Structured traces (Langfuse / OTEL)
04

Hyper-Growth: The Production Runtime

The metaphor now is a hospital emergency room. Multiple teams, triage protocols, escalation paths. If one doctor is unavailable, a patient doesn't die — someone else picks it up. If a procedure fails, there's a documented recovery protocol. Nothing is ad-hoc.

Let's build this layer by layer, in order of impact.

Fig 2 — Production agent runtime. Workers are stateless; state is external; failures have defined paths.

Layer 1 — Async task queue + persistent state

The first move is to make agent runs async and resumable. When a user triggers an agent, you don't run it inline — you enqueue a task. A Celery worker picks it up, and checkpoints progress to Redis/Postgres after every step.

agent_task.py Python 
from celery import Celery
import redis, json, uuid

app = Celery("agent", broker="redis://localhost:6379/0")
r = redis.Redis(host="localhost", port=6379)

@app.task(
    bind=True,
    max_retries=3,
    default_retry_delay=30,  # seconds
    acks_late=True,           # ack only after success, not on pickup
)
def run_agent_task(self, run_id: str, task: str):
    # Resume from checkpoint if exists
    state_key = f"agent:run:{run_id}"
    raw = r.get(state_key)
    state = json.loads(raw) if raw else {"messages": [], "step": 0}

    try:
        result = run_agent_loop(state, task, checkpoint_fn=lambda s: r.setex(
            state_key, 3600, json.dumps(s)
        ))
        r.delete(state_key)       # cleanup on success
        return {"status": "success", "result": result}

    except RateLimitError as e:
        # Transient — retry with backoff
        raise self.retry(exc=e, countdown=exponential_backoff(self.request.retries))

    except PermanentToolError as e:
        # Not retriable — send to DLQ with full context
        send_to_dlq(run_id, task, state, error=str(e))
        return {"status": "failed", "error": str(e)}


def exponential_backoff(retry_count: int) -> int:
    return min(2 ** retry_count * 10, 300)  # 10s, 20s, 40s, cap at 300s
Why acks_late=True matters

With default Celery config, a task is acknowledged (removed from queue) as soon as a worker picks it up. If the worker crashes mid-run, the task is lost. acks_late=True means the task stays in the queue until it completes or explicitly fails — same mental model as "at-least-once delivery" in Kafka.

Layer 2 — Circuit breakers for tool calls

A circuit breaker is the most important pattern you can add to a production agent. The mental model: if Jira's API has been failing for the last 30 seconds, stop calling Jira. Wait. Let it recover. Don't hammer a sick service.

circuit_breaker.py Python 
from pybreaker import CircuitBreaker, CircuitBreakerError
import redis

# One breaker per external dependency
jira_breaker = CircuitBreaker(
    fail_max=5,            # trip after 5 failures
    reset_timeout=60,      # try again after 60s
    name="jira",
)
llm_breaker = CircuitBreaker(
    fail_max=3,
    reset_timeout=30,
    name="openai",
)

@jira_breaker
def create_jira_ticket_safe(idempotency_key: str, **kwargs):
    # Check idempotency: have we already created this?
    cache_key = f"jira:created:{idempotency_key}"
    if r.get(cache_key):
        return json.loads(r.get(cache_key))   # return cached result

    result = jira_client.create_issue(**kwargs)

    # Cache success for 24h to prevent duplicates on retry
    r.setex(cache_key, 86400, json.dumps(result))
    return result

def execute_tool_safe(name: str, args: dict, run_id: str, call_id: str):
    idempotency_key = f"{run_id}:{call_id}"   # stable per agent run + call
    try:
        if name == "create_jira_ticket":
            return create_jira_ticket_safe(idempotency_key, **args)
    except CircuitBreakerError:
        # Jira is down — return a graceful degradation response
        return {
            "status": "degraded",
            "message": "Jira is temporarily unavailable. Ticket queued.",
            "queued_id": idempotency_key,
        }
The idempotency key pattern

Construct the key as {run_id}:{call_id} where call_id is the LLM's tool call ID (stable across retries of the same step). This guarantees that if the agent retries after a network error mid-Jira-call, you don't create a duplicate ticket.

Layer 3 — Retry budget + dead-letter queue

Not all failures are equal. A transient failure (network blip, rate limit) deserves a retry. A permanent failure (malformed input, authorization error, max context exceeded) should never be retried — it wastes resources and masks bugs. Distinguish them explicitly.

failure_classification.py Python 
from openai import RateLimitError, APITimeoutError, BadRequestError

TRANSIENT_ERRORS = (
    RateLimitError,     # 429 — back off and retry
    APITimeoutError,    # timeout — retry with longer timeout
    ConnectionError,    # network — retry
)

PERMANENT_ERRORS = (
    BadRequestError,    # 400 — malformed, fix the code
    AuthenticationError, # 401 — invalid key
    ContextWindowError, # custom — too many tokens
)

def classify_and_handle(error: Exception, run_id: str, state: dict):
    if isinstance(error, TRANSIENT_ERRORS):
        raise  # let Celery retry with backoff

    if isinstance(error, PERMANENT_ERRORS):
        send_to_dlq(
            run_id=run_id,
            error_type=type(error).__name__,
            error_msg=str(error),
            state_snapshot=state,   # full context for debugging
            requires_human=True,
        )
        return  # don't retry

def send_to_dlq(run_id, error_type, error_msg, state_snapshot, requires_human):
    # Persist to Postgres DLQ table
    db.execute("""
        INSERT INTO agent_dlq (run_id, error_type, error_msg,
                               state_snapshot, requires_human, created_at)
        VALUES (%s, %s, %s, %s, %s, NOW())
    """, [run_id, error_type, error_msg,
             json.dumps(state_snapshot), requires_human])

    # Alert on-call if human review needed
    if requires_human:
        pagerduty.trigger(
            summary=f"Agent DLQ: {error_type} on run {run_id}",
            severity="warning",
        )

Layer 4 — Scaling infrastructure

At 50,000 runs/day, your primary scaling lever is queue depth → worker count. Kubernetes HPA can scale Celery workers based on the Redis/SQS queue depth metric. But there are two agent-specific scaling constraints most teams miss:

Constraint Why it bites you Mitigation
LLM rate limits OpenAI TPM limits hit before your CPU does. Adding more workers makes it worse. Token bucket per-model. Slow down enqueue rate, not worker count.
Tool API rate limits Jira: 300 req/min. Slack: 1 req/sec per endpoint. Agents amplify this. Per-tool rate limiting with Redis leaky bucket. Circuit breakers as backstop.
Context window cost Long-running agents accumulate massive message history. Cost scales with steps². Summarize past N steps into a rolling summary. Keep window bounded.
Cold start latency Spinning up a new worker takes 15-30s. Bad for p99 if queue spikes. Keep a warm pool of 2-3 min workers. HPA for burst above baseline.

The Golden Rule

Here's the mental model that will serve you across every agent project: an AI agent is a long-running distributed transaction.

In traditional backend systems, you handle long-running distributed transactions with sagas — a sequence of local transactions where each step has a corresponding compensation action if something downstream fails. Agents need the same thing:

compensation_example.py Python 
# Saga pattern for agents: each action has a compensation
COMPENSATIONS = {
    "create_jira_ticket": "delete_jira_ticket",
    "send_slack_message": "delete_slack_message",  # within 60s
    "update_confluence":  "revert_confluence_page",
}

def compensate_run(run_id: str, completed_actions: list):
    """Roll back side effects if the run fails after partial completion."""
    for action in reversed(completed_actions):
        comp_fn = COMPENSATIONS.get(action["tool"])
        if comp_fn:
            execute_tool_safe(
                name=comp_fn,
                args={"id": action["result"]["id"]},
                run_id=run_id,
                call_id=f"compensation:{action['call_id']}",
            )
            log(f"Compensated {action['tool']} for run {run_id}")

You won't implement full compensation logic on day one — and you don't need to. But the moment your agent takes real-world side effects (creates tickets, sends emails, charges users), you need to think about what "undo" looks like.

⚠ Common Mistakes & Gotchas
  1. Retrying non-idempotent tools blindly. Celery will retry your task. If create_jira_ticket doesn't have an idempotency key, you'll create N tickets per retry. Always check before you create.
  2. Using the same circuit breaker for all LLM models. GPT-4o and GPT-3.5 have different rate limits and reliability profiles. Use separate breakers per model endpoint.
  3. Treating "max steps reached" as success. It's a silent failure. Log it, alert on it, and put it in the DLQ if the task wasn't completed.
  4. Not bounding context window growth. A 20-step agent with full message history can hit 40K+ tokens per LLM call. Implement rolling summarization after step 10.
  5. Setting retry delays too short. If a tool is rate-limited at 429, retrying in 1 second makes it worse. Use exponential backoff with jitter: base * 2^n + random(0, base).
  6. No human-review path for DLQ items. The DLQ is useless if no one looks at it. Wire it to PagerDuty or Slack, and build a simple admin UI to inspect + replay failed runs.
07

Scale Considerations

At 100 million operations per day (roughly 1,200 ops/sec sustained), the agent runtime itself becomes the distributed system. Here's what changes:

Component 50k runs/day (current) 100M ops/day (hyper-growth)
Queue Redis + Celery, single region SQS FIFO per priority tier, multi-region
State store Postgres + Redis Postgres (sharded by org_id) + Redis Cluster
Workers k8s Deployment, HPA on queue depth k8s with KEDA (event-driven autoscaler), spot + on-demand mix
LLM routing Direct OpenAI API LLM gateway (LiteLLM / Portkey) with model fallbacks + cost routing
Observability Langfuse, single instance OTEL collector → Datadog / Honeycomb, with sampling at 10%
DLQ Postgres table + PD alert Dedicated DLQ service with auto-replay for transient failures

The most important hyper-growth architectural decision is priority queues. Not all agent runs are equal. An agent triggered by a paying enterprise customer at $500/month should not wait behind a background batch job. SQS FIFO with message groups lets you implement strict FIFO within a priority tier while processing high-priority messages first.

The LLM gateway pattern

At 100M ops/day you will hit provider-level limits regardless of your account tier. An LLM gateway (LiteLLM or Portkey) sits between your agents and the providers. It handles model fallbacks (GPT-4o → Claude → Gemini), per-tenant rate limiting, cost tracking per team/product, and automatic retry with provider switching. This is table stakes at scale.
