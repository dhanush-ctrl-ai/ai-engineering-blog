Chapter 02

LLM Eval Harnesses in CI/CD

TL;DR

  • Most AI teams discover regressions in production. A CI eval harness catches them in the PR.
  • Day One: a pytest script with golden Q&A pairs and cosine similarity scoring.
  • Production: async eval workers, per-test latency budgets, cost tracking per PR, and a blocking GitHub Actions step.
  • The golden dataset is your most valuable artefact - build it from real user queries, not synthetic ones.
  • Key metrics: answer faithfulness, retrieval precision@k, latency p95, token cost per query.

The moment you realise you need this

You update your prompt template on a Monday. Tests pass - there are no tests. Deploy goes out. By Wednesday, three engineers have filed bugs: the agent is now hallucinating Jira ticket numbers it used to get right. You roll back, but you have lost two days and your users' trust.

This is the most common failure mode in AI systems: silent regression. Unlike traditional software where a wrong function return is deterministic and testable, an LLM that used to answer correctly can start answering differently after a model update - and no unit test catches it unless you deliberately measure output quality.

An eval harness is a test suite for correctness, not just execution. It answers: "did the agent do the right thing?" not just "did it run without crashing?"

What you are actually measuring

Before writing any code, decide what "correct" means for your agent. There are four dimensions worth tracking in CI:

MetricWhat it measuresHow to score itFail threshold
FaithfulnessDoes the answer come from retrieved context, not hallucination?LLM-as-judge: ask Claude/GPT-4 to rate 0-1< 0.85
Answer relevanceDoes the answer actually address the question?Cosine similarity of question vs answer embeddings< 0.75
Retrieval precision@kAre the top-k retrieved chunks the right ones?Overlap with manually labelled ground-truth chunks< 0.60
Latency p95Is the agent fast enough?Wall-clock time across N runs> 30s
Token costDid this PR make the agent more expensive?tiktoken count x model price> 20% increase

You do not need all five on day one. Start with faithfulness and answer relevance - they catch 80% of regressions.

Day One: the pytest golden set

The metaphor here is a spell-checker for your agent. You give it a fixed dictionary of right answers; it flags anything that deviates. Simple, fast, enough to start.

Building the golden dataset

The most common mistake is generating synthetic Q&A pairs with GPT-4. Do not. Use real queries from real users - look at your agent's logs, Slack DMs, support tickets. 50 real questions beat 500 synthetic ones.

The Day One eval script

# test_agent_eval.py
import pytest, json, time, math
from openai import OpenAI

client = OpenAI()

def embed(text):
    return client.embeddings.create(model="text-embedding-3-small", input=text).data[0].embedding

def cosine(a, b):
    dot = sum(x*y for x,y in zip(a,b))
    return dot / (math.sqrt(sum(x**2 for x in a)) * math.sqrt(sum(x**2 for x in b)))

with open("golden_set.json") as f:
    GOLDEN = json.load(f)

@pytest.mark.parametrize("case", GOLDEN, ids=[c["id"] for c in GOLDEN])
def test_agent_answer_quality(case, agent):
    start = time.time()
    result = agent.run(case["question"])
    elapsed = time.time() - start

    # Latency check
    assert elapsed < case["max_latency_s"], \
        f"Latency {elapsed:.1f}s exceeded budget {case['max_latency_s']}s"

    # Keyword coverage
    answer_lower = result.answer.lower()
    missing = [kw for kw in case["expected_keywords"] if kw not in answer_lower]
    assert not missing, f"Missing keywords: {missing}"

    # Semantic relevance
    sim = cosine(embed(case["question"]), embed(result.answer))
    assert sim > 0.70, f"Answer relevance too low: {sim:.2f}"

Production: CI/CD integration with GitHub Actions

The Day One script runs on your laptop. The production eval harness runs on every PR - automatically, blocking merge if scores drop.

# .github/workflows/eval.yml
name: Agent Eval
on:
  pull_request:
    branches: [main]

jobs:
  eval:
    runs-on: ubuntu-latest
    timeout-minutes: 20
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: "3.11"
      - run: pip install -r requirements.txt
      - name: Run eval harness
        env:
          OPENAI_API_KEY: ${{ secrets.OPENAI_API_KEY }}
        run: |
          pytest tests/eval/ \
            --eval-threshold-faithfulness=0.85 \
            --eval-threshold-relevance=0.75 \
            --json-report --json-report-file=eval-report.json

Async eval workers for speed

Running 50 eval cases sequentially takes 5-10 minutes in CI. Use asyncio.gather to run them in parallel - 50 cases finish in the time of the slowest one:

# tests/eval/runner.py
import asyncio, time
from dataclasses import dataclass

@dataclass
class EvalResult:
    case_id: str
    faithfulness: float
    relevance: float
    latency_s: float
    cost_usd: float
    passed: bool

async def eval_case(agent, case):
    start = time.perf_counter()
    result = await agent.arun(case["question"])
    latency = time.perf_counter() - start

    faithfulness = await score_faithfulness(
        question=case["question"],
        answer=result.answer,
        context="\n".join(n.text for n in result.source_nodes)
    )
    relevance = cosine_similarity(embed(case["question"]), embed(result.answer))
    cost = estimate_cost(result.token_usage)

    passed = faithfulness >= 0.85 and relevance >= 0.75 and latency < case.get("max_latency_s", 30)
    return EvalResult(case["id"], faithfulness, relevance, latency, cost, passed)

async def run_all(agent, golden_set):
    return await asyncio.gather(*[eval_case(agent, c) for c in golden_set])

⚠ Common Mistakes & Gotchas

  1. Using synthetic golden data. GPT-4 generated Q&A pairs test whether your agent behaves like GPT-4, not whether it helps your users. Mine real query logs.
  2. LLM-as-judge with the same model you are evaluating. If your agent uses GPT-4o and your judge is GPT-4o, the judge will systematically agree. Use a different model or embedding-based scoring.
  3. Threshold too strict on day one. Setting faithfulness at 0.95 with 5 test cases means you will be debugging the eval harness instead of the agent. Start at 0.80, tighten as your golden set grows.
  4. Not tracking cost delta. A PR can pass all quality checks but double your token spend. Always report cost per query as a CI metric.
  5. Skipping eval in hotfix PRs. That is exactly when regressions sneak through.
