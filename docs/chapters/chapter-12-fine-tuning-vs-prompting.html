Chapter 12

Fine-Tuning vs Prompting: When and How to Fine-Tune

TL;DR

  • Fine-tuning is not a silver bullet. It solves a specific problem: teaching a model a consistent format, style, or domain vocabulary that prompting cannot reliably achieve.
  • Day One: try prompting first. Always. Fine-tuning has a high upfront cost and a hidden maintenance cost that most teams underestimate.
  • Fine-tune when: you need consistent output format that prompting produces unreliably, your system prompt is bloated beyond 4k tokens with examples, or latency is critical and you must shorten the context.
  • Do not fine-tune when: you need factual knowledge (use RAG), you have fewer than 500 high-quality examples, or your task changes frequently.
  • Eval-driven fine-tuning: always establish a baseline eval score before fine-tuning, and only ship the fine-tuned model if it beats the baseline on the test set.

The most common fine-tuning mistake

An engineering team notices their agent sometimes returns answers in the wrong format - occasionally in markdown, occasionally in plain text, occasionally with bullet points, occasionally without. They spend three weeks fine-tuning GPT-4o-mini on 200 examples of "correct" format and deploy the result. Output format is now consistent. But answer quality has dropped 15%, the model now occasionally hallucinates domain terms it "learned" during fine-tuning, and every time the format requirements change they need another fine-tuning run.

The correct fix was a better system prompt with explicit format instructions and a few-shot example. It would have taken 30 minutes.

Fine-tuning is the last resort, not the first. If you have not exhausted prompting strategies - few-shot examples, chain-of-thought, output format constraints - you are not ready to fine-tune.

Decision framework: prompt vs fine-tune

Day One: few-shot prompting as fine-tuning substitute

# Before reaching for fine-tuning, exhaust few-shot prompting
SYSTEM_PROMPT = """You are an engineering wiki assistant.

Always respond in this EXACT format:
---
ANSWER: [one paragraph, max 150 words, no markdown]
SOURCE: [filename without path]
CONFIDENCE: [HIGH / MEDIUM / LOW]
---

Examples of correct format:

User: How do I restart the payment service?
Assistant:
---
ANSWER: Run kubectl rollout restart deployment/payment-svc in the infra cluster.
Wait 2 minutes for pods to cycle. Check pod health with kubectl get pods -l app=payment-svc.
SOURCE: kubernetes-runbooks.md
CONFIDENCE: HIGH
---

User: What is the SLA for P2 incidents?
Assistant:
---
ANSWER: P2 incidents require acknowledgment within 30 minutes and resolution within 4 hours.
The on-call engineer must post an update in #incidents every 60 minutes until resolved.
SOURCE: incident-response-policy.md
CONFIDENCE: HIGH
---"""

If two or three well-chosen few-shot examples achieve the format consistency you need, you have saved weeks of fine-tuning work. Only proceed to fine-tuning if this approach fails after iteration.

Production fine-tuning: the eval-driven workflow

Fine-tuning without an eval harness is guesswork. You have no way to know whether the fine-tuned model is better or worse than the base model on your actual use case. The workflow must be: baseline eval → collect training data → fine-tune → eval on held-out test set → deploy only if it beats baseline.

LoRA fine-tuning on open-source models

# For Llama 3.1, Mistral, Qwen2 - fine-tune locally with LoRA
from peft import LoraConfig, get_peft_model
from transformers import AutoModelForCausalLM, AutoTokenizer, TrainingArguments
from trl import SFTTrainer

model_name = "meta-llama/Llama-3.1-8B-Instruct"
model = AutoModelForCausalLM.from_pretrained(model_name, load_in_4bit=True)

# LoRA: only train ~0.5% of weights, but adapt behaviour significantly
lora_config = LoraConfig(
    r=16,               # rank - higher = more capacity, more VRAM
    lora_alpha=32,      # scaling factor
    target_modules=["q_proj","v_proj"],  # which layers to adapt
    lora_dropout=0.05,
    bias="none",
)
model = get_peft_model(model, lora_config)
model.print_trainable_parameters()

⚠ Common Mistakes & Gotchas

  1. Fine-tuning on AI-generated examples. If your training data was generated by GPT-4 and then lightly edited, you are teaching a model to imitate GPT-4 - not to actually answer your users' questions well. Use real user interactions, human-verified answers.
  2. No held-out test set. If you evaluate on training data, you will always see improvement. The test set must be completely separate from training data and must not influence hyperparameter choices.
  3. Forgetting catastrophic forgetting. Fine-tuning on a narrow dataset can degrade the model's general capabilities. Always run a broad eval, not just task-specific evals, on the fine-tuned model before deploying.
  4. Treating fine-tuning as a one-time event. Your data distribution drifts over time. A fine-tuned model trained on 6-month-old data will gradually fall behind a well-prompted base model with an updated system prompt. Plan for periodic re-fine-tuning.
  5. Fine-tuning instead of fixing your data pipeline. If the model gives wrong answers because your RAG pipeline retrieves the wrong chunks, fine-tuning will not fix it. It will just make the model more confidently wrong. Fix the root cause first.
