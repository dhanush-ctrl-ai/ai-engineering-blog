Chapter 05

RAG at Scale: From a Single PDF to 100M Documents

TL;DR

  • Five bottlenecks break RAG at scale: ingestion speed, embedding throughput, vector DB write limits, retrieval latency, and reranker cost.
  • Day One: sequential ingestion, in-memory vector store, top-k cosine search. Works for fewer than 1k docs.
  • Production: parallel async ingestion workers, batched GPU embeddings, sharded vector DB, HNSW ANN index, two-stage retrieve-then-rerank.
  • The ingestion pipeline is a distributed system. Design it like one.
  • At 100M docs: shard by org_id, use a dedicated embedding service, and separate hot vs cold index tiers.

Where naive RAG breaks

The LlamaIndex RAG series covered building a correct pipeline. This chapter covers building a fast one. Five distinct bottlenecks appear in order as you scale - fix them in this sequence or you will optimise the wrong thing.

Bottleneck 1: Sequential ingestion - async workers

The naive pipeline processes one document at a time: read - chunk - embed - store. At 50,000 docs with 2 seconds per doc, that is 28 hours. The fix is to parallelise across all four stages independently.

# ingestion_pipeline.py
import asyncio
from llama_index.core.ingestion import IngestionPipeline
from llama_index.core.node_parser import SentenceSplitter
from llama_index.embeddings.huggingface import HuggingFaceEmbedding

embed_model = HuggingFaceEmbedding(
    model_name="BAAI/bge-large-en-v1.5",
    device="cuda",
    embed_batch_size=256,
)

pipeline = IngestionPipeline(
    transformations=[SentenceSplitter(chunk_size=512, chunk_overlap=64), embed_model],
    vector_store=pinecone_store,
    docstore=redis_docstore,
    docstore_strategy="upserts_and_delete",  # deduplication
)

async def ingest_batch(docs, semaphore):
    async with semaphore:
        await pipeline.arun(documents=docs, num_workers=4)

async def ingest_all(all_docs, batch_size=500):
    sem = asyncio.Semaphore(8)   # max 8 concurrent batches
    batches = [all_docs[i:i+batch_size] for i in range(0, len(all_docs), batch_size)]
    await asyncio.gather(*[ingest_batch(b, sem) for b in batches])

Bottleneck 4: Retrieval latency - HNSW ANN index

Exact cosine search is O(N) - at 1M vectors it takes 2-5 seconds. HNSW (Hierarchical Navigable Small World) approximate nearest neighbour search reduces this to O(log N): milliseconds at any scale, with a small accuracy trade-off you control via ef_search.

# Qdrant with explicit HNSW config
from qdrant_client import QdrantClient
from qdrant_client.models import VectorParams, Distance, HnswConfigDiff

client = QdrantClient(url="http://localhost:6333")
client.create_collection(
    collection_name="wiki_chunks",
    vectors_config=VectorParams(size=1024, distance=Distance.COSINE),
    hnsw_config=HnswConfigDiff(
        m=16,              # edges per node - higher = better recall, more RAM
        ef_construct=200,  # build-time quality
    ),
)

# At query time: ef (search width) trades accuracy for speed
results = client.search(
    collection_name="wiki_chunks",
    query_vector=query_embedding,
    limit=20,
    search_params={"hnsw_ef": 128},
)

⚠ Common Mistakes & Gotchas

  1. Using OpenAI embeddings at bulk scale. At 100M chunks, OpenAI's embedding API rate limit makes bulk ingestion take weeks. Use a local GPU embedding model (BAAI/bge) for ingestion; OpenAI only for query-time.
  2. Not deduplicating on re-ingestion. Without DocstoreStrategy.UPSERTS_AND_DELETE, every daily re-run doubles your index size. Hash document content and skip unchanged docs.
  3. Building HNSW index incrementally one-by-one. Adding vectors one at a time is 10x slower than bulk loading. Always bulk-load, then serve. Add incrementally for daily updates but rebuild the full index weekly.
  4. Reranking all retrieved candidates. Running a cross-encoder over top-100 at 10k queries/min is prohibitive GPU cost. Retrieve top-20 with ANN, rerank to top-5. That ratio is the sweet spot.
  5. Single-shard vector DB beyond 10M vectors. Most hosted vector DBs hit latency spikes beyond 10M vectors in a single collection. Shard by team or department before you hit the limit.
